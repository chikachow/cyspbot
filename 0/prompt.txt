### Token Policy

Installation Token Issuance is allowed only when the normalized installation token request matches an explicit checked-in Token Policy rule:

- the caller is a verified principal from a configured Trusted OIDC Issuer
- the signed subject token audience is `cyspbot`
- if the OIDC token has an `azp` claim, that claim matches `cyspbot`
- GitHub Actions rules require `event_name` listed by the matching rule, OIDC subject context `ref`, `ref_type=branch`, parsed subject repository consistency with signed repository claims, and exact `repository`, `ref`, parsed subject ref, and `workflow_ref` matches
- Fly Machine rules require an a<truncated omitted_approx_tokens="513" />y key. Name-only Fly rules are supported only as an explicit compatibility mode because Fly names are easier to rename or reuse than IDs. Optional machine-level narrowing follows the selected identity mode: ID rules narrow by `machine_id`, name rules narrow by `machine_name`, and ID-and-name rules narrow by both.

Fly Machine principal derivation requires these verified token claims: `sub`, `exp`, `nbf`, `org_id`, `org_name`, `app_id`, `app_name`, `machine_id`, `machine_name`, and `machine_version`. The `sub` claim must exactly equal `org_name:app_name:machine_name`. cyspbot also accepts optional Fly runtime claims `region`, `image`, `image_digest`, and `image_tag` for policy logging.

Fly Machine support is deny-by-default. A verified Fly Machine token is accepted for exchange only when a checked-in Token Policy rule explicitly allows that Fly principal, repository resource, and permission request.
