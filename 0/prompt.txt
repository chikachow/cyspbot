## What the CEL library already does

The installed `@bufbuild/cel` 0.6.0 is current. Its intended reusable-evaluation workflow is `celEnv()` → `parse()` → `plan()`, and `plan()` handles CEL comprehensions and lexical variables itself. [Official package documentation](https://www.npmjs.com/package/@bufbuild/cel), [official example](https://raw.githubusercontent.com/bufbuild/cel-es/main/packages/example/src/example.ts).

Direct experiments against the installed package confirmed:

- `plan()` correctly handles `.exists(group, ...)` and its local variables.
- Unknown variables evaluate to a `CelError` (`unresolved attribute`).
- Missing claims and type mismatches produce `CelError`.
- Non-boolean expressions return their value and can be rejected with `result === true`.
- The library accepts ordinary objects, arrays, strings, numbers, booleans, and null as `CelInput`.

The package even contains `check()` and `outputType()` implementations that understand scopes, identifiers, and result types—but they are not exported by the package’s public interface. The current upstream `index.ts` still omits them. [Current upstream exports](https://raw.githubusercontent.com/bufbuild/cel-es/main/packages/cel/src/index.ts).

## Complexity that can be deleted

### 1. Delete the custom CEL AST walker

Approximately 85 lines implement:

- recursive protobuf AST traversal;
- external-root detection;
- comprehension lexical scope;
- `iterVar`, `iterVar2`, and accumulator handling;
- protobuf-node shape guards.

This is brittle against changes to CEL’s parsed-expression representation and duplicates knowledge already held by the library’s internal checker.

The only behavior it adds is rejecting unknown external identifiers at startup. Without it, an identifier such as `typo` produces a fail-closed `CelError` during evaluation. That is an availability/configuration diagnostic, not an authorization boundary.

Given that policy is checked in and every production rule has a positive evaluation test, I would accept that tradeoff and remove the walker.

### 2. Delete most of `jsonLikeValue`

JWT claims are JSON values. The converter currently supports or transforms:

- `bigint`;
- `Uint8Array`;
- symbols;
- functions;
- arbitrary objects;
- undefined nested values.

Those cannot come from a decoded JWT JSON payload. Supporting them creates branches, tests, and semantics outside the domain.

A shallow normalization is sufficient:

```ts
const claims = Object.fromEntries(
  Object.entries(input.subjectToken.claims).filter(
    (entry): entry is [string, CelInput] => entry[1] !== undefined,
  ),
);
```

Alternatively, after authentication has established that claims came from decoded JSON, use one documented cast to `Record<string, CelInput>`. The existing recursive conversion provides little business value.

### 3. Remove `isCelError`

This:

```ts
return !isCelError(result) && result === true;
```

is equivalent to:

```ts
return result === true;
```

A `CelError`, false, null, number, or any other value all deny.

### 4. Keep compilation and caching

The `WeakMap` and reusable planned evaluator are earning their keep. Replacing them with `run()` would reduce a few lines but reparse and replan CEL on every token request. The library itself recommends `parse()` plus `plan()` for reusable evaluation.

The fail-closed `try/catch` is also worth retaining. `run()` promises not to throw, but the public documentation does not make the same explicit guarantee for planned evaluators or caller-provided binding construction.

type Evaluator = ReturnType<typeof plan>;
const evaluators = new WeakMap<TokenPolicyRule, Evaluator>();

export function tokenPolicyConditionIsValid(rule: TokenPolicyRule): boolean {
  return evaluatorFor(rule) !== null;
}

export function tokenPolicyConditionMatches(
  rule: TokenPolicyRule,
  input: TokenPolicyInput,
): boolean {
  try {
    return evaluatorFor(rule)?.(bindingsFor(input)) === true;
  } catch {
    return false;
  }
}

function evaluatorFor(rule: TokenPolicyRule): Evaluator | null {
  const cached = evaluators.get(rule);
  if (cached !== undefined) return cached;

  try {
    const evaluator = plan(env, parse(rule.when));
    evaluators.set(rule, evaluator);
    return evaluator;
  } catch {
    return null;
  }
}

`bindingsFor()` then needs only to assemble `claims`, `subject`, and `request`, not recursively reinterpret JWT data.
